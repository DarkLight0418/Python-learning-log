# VDS 아키텍처 초안

## 1. 아키텍처 목표
VDS의 아키텍처는 자료구조 연산 규칙, Scene 생성 책임, Scene 렌더링 책임을 분리하여 다음 목표를 달성한다.

- 도메인 규칙의 순수성 유지
- Scene 중심 출력 계약 유지
- View의 자료구조 비의존성 확보
- 테스트 시뮬레이터와 GUI 입력 경로의 공존

---

## 2. 핵심 구조

### 2.1 흐름
입력(시뮬레이터 / GUI) → 도메인 연산 → Presenter 변환 → Scene 생성 → View 렌더링

### 2.2 레이어 개요
- **App / Runner**: 입력을 받아 적절한 도메인 연산과 Presenter 흐름을 호출한다.
- **Domain(Core)**: 자료구조 규칙과 상태를 관리한다.
- **Presenter**: 도메인 상태 또는 연산 결과를 Scene으로 변환한다.
- **View**: Scene만 받아 렌더링한다.

---

## 3. 레이어별 책임

### 3.1 Domain
책임:
- 자료구조 상태 보관
- 연산 규칙 보장(push/pop/top 등)
- 도메인 예외 발생

비책임:
- 화면 표시
- 좌표 계산
- 색상/스타일 결정
- UI 프레임워크 의존

### 3.2 Presenter
책임:
- 도메인 상태를 Scene으로 변환
- 연산 결과에 따른 강조 정보 부여
- SceneMeta, Drawable 구성
- 오류 상황을 Scene 또는 출력 상태로 표현하는 정책 수행

비책임:
- 실제 캔버스 그리기
- UI 이벤트 직접 처리
- 자료구조 내부 규칙 구현

### 3.3 View
책임:
- Scene 렌더링
- Drawable 타입별 그림 생성
- 필요 시 화면 갱신/애니메이션 연결

비책임:
- 자료구조 규칙 판단
- 도메인 상태 변경
- 도메인 예외 해석

### 3.4 App / Runner
책임:
- GUI 사용자 입력 또는 시뮬레이터 입력 해석
- 적절한 도메인 연산 호출
- Presenter를 통해 Scene 생성 요청
- 실행 흐름 조합

---

## 4. 핵심 개념 모델

### 4.1 Step
- 단일 연산 또는 단일 실행 단계
- 예: push(10), pop(), clear()

### 4.2 Scene
- 특정 시점의 시각화 가능한 장면
- View가 렌더링 가능한 출력 단위

### 4.3 Sequence
- Step과 Scene의 순차적 연결
- 재생, 리플레이, 단계 이동 등 확장 기반

---

## 5. Scene 설계 원칙
- Scene은 View가 자료구조를 몰라도 렌더링 가능해야 한다.
- Scene은 좌표, 크기, 텍스트, 스타일 등 표시 필요한 정보를 충분히 포함해야 한다.
- Scene은 메타 정보(title, complexity, description 등)를 포함할 수 있다.
- Scene 내 Drawable은 안정적 식별자(key)를 가질 수 있어야 한다.

---

## 6. 오류 처리 전략
오류 처리는 두 레벨로 나눈다.

### 6.1 도메인 레벨
- 잘못된 연산에 대해 예외를 발생시킨다.
- 예: EmptyStackError

### 6.2 표현 레벨
- 오류를 사용자/시뮬레이터가 이해 가능한 출력으로 변환한다.
- 필요 시 오류 Scene으로 표현할 수 있다.

권장 방향:
- 도메인 규칙 위반은 예외로 유지
- Presenter 또는 Runner에서 오류를 Scene 또는 에러 출력으로 변환

---

## 7. 확장 전략
현재는 Stack을 우선 대상으로 하지만, 다음 방향으로 확장 가능하도록 둔다.

- Queue / List / Heap 등 다른 자료구조 추가
- 자료구조별 Presenter 추가
- 공통 Scene 모델 재사용
- View 재사용 및 최소 변경

---

## 8. 설계 원칙 요약
- Domain은 UI를 모른다.
- View는 자료구조를 모른다.
- Presenter는 도메인과 View 사이의 번역자 역할을 맡는다.
- Scene은 제품의 핵심 출력 계약이다.
- step / scene / sequence 구조는 향후 애니메이션 및 리플레이 확장의 기반이다.
