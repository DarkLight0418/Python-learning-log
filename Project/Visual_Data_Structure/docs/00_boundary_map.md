# VDS 경계 확정 문서

## 목적
이 문서는 Visual Data Structure(VDS) 프로젝트에서 **요구사항 / 설계 / 구현**의 경계를 분리하기 위한 기준 문서이다. 이 문서의 목적은 다음과 같다.

- 요구사항 문서에 설계 세부가 섞이는 것을 방지한다.
- 설계 문서에 구현 세부가 과도하게 들어가는 것을 방지한다.
- 리팩토링 시 "무엇이 바뀐 것인지"를 정확히 분류할 수 있게 한다.

---

## 1. 분류 기준

### 1.1 요구사항
**시스템이 외부에서 보았을 때 반드시 해야 하는 일**

판별 질문:
- 사용자나 테스트 시뮬레이터가 관찰할 수 있는가?
- 구현 방식이 바뀌어도 유지되는가?
- "무엇을 해야 하는가"로 설명되는가?

예:
- 시스템은 GUI 사용자 입력으로 자료구조 연산을 실행할 수 있어야 한다.
- 시스템은 테스트 시뮬레이터로 연산 시퀀스를 실행할 수 있어야 한다.
- 시스템은 각 연산 결과를 Scene 프레임으로 제공해야 한다.
- 시스템은 잘못된 연산 요청에 대해 실패를 명확히 알려야 한다.

### 1.2 설계
**요구사항을 만족시키기 위해 책임을 어떻게 분리할 것인가**

판별 질문:
- 어떤 계층이 무엇을 알아야 하는가?
- 어떤 객체가 어떤 책임을 가져야 하는가?
- 변화 가능성이 큰 정책을 어디에 격리할 것인가?

예:
- Domain은 자료구조 규칙만 담당한다.
- Presenter는 도메인 상태를 Scene으로 변환한다.
- View는 Scene 렌더링만 담당한다.
- View는 자료구조 종류를 알지 않는다.
- step / scene / sequence 단위로 모델을 확장 가능하게 둔다.

### 1.3 구현
**설계를 실제 코드/파일/프레임워크로 옮긴 결과물**

판별 질문:
- 클래스명, 함수명, 파일명 수준의 선택인가?
- 특정 라이브러리/프레임워크에 묶여 있는가?
- 리팩토링으로 얼마든지 바뀔 수 있는가?

예:
- `src/vds/core/stack.py`
- `TkCanvasView`
- `SceneBuilder` 클래스 분리
- `registry[(RectItem, RectItem)] = rect_interp`
- `dataclass(frozen=True)` 사용

## 2. VDS 주요 항목 분류표

| 항목 | 분류 | 설명 |
|---|---|---|
| GUI 사용자가 push/pop/top 등을 실행할 수 있어야 함 | 요구사항 | 외부 사용자가 보는 기능 |
| 테스트 시뮬레이터가 연산 시퀀스를 실행할 수 있어야 함 | 요구사항 | 시스템 사용 방식 자체 |
| 각 연산 결과를 Scene 프레임으로 생성해야 함 | 요구사항 | 제품의 핵심 출력 계약 |
| View는 Scene만 받아 렌더링해야 함 | 설계 | 책임 분리 원칙 |
| Domain은 UI를 몰라야 함 | 설계 | 의존 방향 규칙 |
| Presenter가 도메인 상태를 Scene으로 번역함 | 설계 | 변환 책임의 위치 |
| step / scene / sequence 중심으로 모델링함 | 설계 | 시스템 구조화 방식 |
| 오류를 예외만 던질지, 오류 Scene으로 표현할지 | 요구사항 + 설계 | 실패는 요구사항, 처리 방식은 설계 |
| `EmptyStackError` 클래스 분리 | 구현 | 코드 조직 문제 |
| `SceneBuilder` 분리 | 설계 | Presenter 비대화 방지용 책임 분리 |
| `AnimationEngine` 분리 | 설계 | 변화 가능성이 큰 정책 분리 |
| `canvas_view.py` 내부 draw registry 구조 | 구현 | 구체적 코드 구조 |
| `core / presenter / view / app` 폴더 구조 | 구현 | 설계의 물리적 배치 |
| `snapshot()` 메서드 제공 | 요구사항 + 설계 | 외부 계약이면 요구사항, 내부 정책이면 설계 |
| SceneMeta에 title/complexity 포함 | 요구사항 + 설계 | 사용자에게 보여주면 요구사항, 채우는 위치는 설계 |
| Tkinter 사용 | 구현 | 기술 선택 |

## 3. 경계가 헷갈리는 항목 정리

### 3.1 오류 처리
- 요구사항: 시스템은 잘못된 연산 요청에 대해 명확히 실패를 알려야 한다.
- 설계: 실패를 예외로 던질지, Result 객체로 감쌀지, 오류 Scene으로 만들지 결정한다.
- 구현: `EmptyStackError`, `DomainError`, `ValidationError` 등의 클래스 정의 및 위치를 정한다.

### 3.2 Scene
- 요구사항: 시스템은 단계별 결과를 장면(Scene) 단위로 제공해야 한다.
- 설계: Scene이 어떤 정보를 갖고, Drawable이 어떤 종류로 나뉘는지 정한다.
- 구현: `Scene`, `RectItem`, `TextItem`를 어떤 모듈에 어떤 형태로 둘지 정한다.

### 3.3 폴더 구조
- 요구사항 문서에는 폴더 구조를 쓰지 않는다.
- 설계 문서에는 책임 분리 원칙만 쓴다.
- 구현 문서에서만 실제 폴더 구조와 파일 배치를 다룬다.

## 4. 앞으로의 판별 규칙
새로운 아이디어나 변경사항이 생길 때 아래 질문으로 먼저 분류한다.

1. **요구사항인가?**  
   → 사용자가 보거나 테스트로 관찰 가능한 시스템 행동인가?

2. **설계인가?**  
   → 책임 분리와 의존 방향을 정하는 결정인가?

3. **구현인가?**  
   → 파일/클래스/라이브러리/코드 스타일 수준의 선택인가?

## 5. 추천 문서 체계
- `requirements.md` : 제품이 무엇을 해야 하는지 기록
- `architecture.md` : 왜 이런 책임 구조로 나누는지 기록
- `implementation-notes.md` : 실제 리팩토링/파일/클래스 계획 기록
