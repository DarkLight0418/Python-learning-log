# GUI 채팅 애플리케이션 설계 문서 초안

## 1. 프로젝트 개요

본 프로젝트는 Python의 TCP Socket을 이용하여 여러 사용자가 실시간으로 메시지를 주고받을 수 있는 GUI 기반 채팅 애플리케이션을 구현하는 것을 목표로 한다.

전체 구조는 **클라이언트-서버 구조**를 사용한다.
서버는 여러 클라이언트의 접속을 관리하고, 한 클라이언트가 보낸 메시지를 다른 클라이언트들에게 전달한다.
클라이언트는 wxPython 기반 GUI를 통해 사용자가 메시지를 입력하고, 서버로부터 받은 메시지를 화면에 출력한다.

## 3. 전체 시스템 구조

```text
[Client A - wxPython GUI]
        │
        │ TCP Socket
        ▼
[Chat Server]
        ▲
        │ TCP Socket
        │
[Client B - wxPython GUI]
```

서버는 여러 클라이언트와 연결된다.
각 클라이언트는 서버에 메시지를 보내고, 서버는 받은 메시지를 전체 클라이언트에게 브로드캐스트한다.

## 5. 핵심 설계 원칙

이 프로젝트는 다음 원칙을 기준으로 설계한다.

```text
GUI는 화면과 사용자 이벤트만 담당한다.
Network는 서버와의 송수신만 담당한다.
Server는 연결 관리와 메시지 중계만 담당한다.
Protocol은 메시지 형식 변환만 담당한다.
```

즉, GUI 코드가 소켓의 내부 처리 방식을 직접 알지 않도록 하고, 네트워크 코드가 wxPython 위젯을 직접 수정하지 않도록 한다.

## 6. 서버 설계

### 6.1 서버의 책임

서버는 다음 기능을 담당한다.

```text
1. 서버 소켓 생성
2. 클라이언트 접속 대기
3. 클라이언트 연결 수락
4. 클라이언트 목록 관리
5. 클라이언트별 수신 스레드 생성
6. 메시지 수신
7. 전체 클라이언트에게 메시지 브로드캐스트
8. 연결 종료된 클라이언트 제거
```

### 6.2 서버 동작 흐름

```text
서버 실행
↓
HOST와 PORT에 bind
↓
listen으로 접속 대기
↓
클라이언트 접속 시 accept
↓
clients 목록에 추가
↓
클라이언트 전용 스레드 생성
↓
메시지 수신
↓
전체 클라이언트에게 broadcast
```

### 6.3 서버 주요 함수

```text
start_server()
- 서버 소켓을 생성하고 클라이언트 접속을 대기한다.

handle_client(client_socket)
- 특정 클라이언트의 메시지를 계속 수신한다.

broadcast(message)
- 수신한 메시지를 모든 클라이언트에게 전송한다.

remove_client(client_socket)
- 연결이 종료된 클라이언트를 목록에서 제거한다.
```

## 7. 클라이언트 네트워크 설계

### 7.1 클라이언트 네트워크의 책임

`client_network.py`는 GUI와 분리되어 서버 통신만 담당한다.

```text
1. 서버 접속
2. 메시지 전송
3. 서버 메시지 수신
4. 연결 종료 처리
```

### 7.2 주요 함수

```text
connect(host, port)
- 서버에 TCP 연결을 요청한다.

send_message(message)
- 서버로 메시지를 전송한다.

receive_loop()
- 서버에서 오는 메시지를 반복해서 수신한다.

close()
- 서버와의 연결을 종료한다.
```

### 7.3 수신 스레드

클라이언트는 서버 메시지를 계속 기다려야 한다.
하지만 이 작업을 GUI 메인 스레드에서 처리하면 화면이 멈출 수 있다.

따라서 메시지 수신은 별도 스레드에서 수행한다.

```text
GUI 메인 스레드
- 버튼 클릭
- 입력창 처리
- 채팅창 출력

수신 스레드
- socket.recv()
- 서버 메시지 수신
- GUI에 메시지 출력 요청
```

## 9. wxPython에서 스레드 처리 방식

wxPython에서도 Tkinter와 마찬가지로 **백그라운드 스레드에서 GUI 위젯을 직접 수정하면 안 된다.**

따라서 다음 중 하나의 방식을 사용한다.

## 대안 A. `wx.CallAfter()` 사용

수신 스레드에서 메시지를 받은 뒤 GUI 갱신 함수를 안전하게 호출한다.

```text
수신 스레드
↓
서버 메시지 수신
↓
wx.CallAfter(append_message, message)
↓
GUI 메인 스레드에서 채팅창 갱신
```

장점:

- wxPython에서 자주 쓰는 방식이다.
- 구조가 비교적 단순하다.
- Queue 없이도 구현 가능하다.

단점:

- 네트워크 코드가 wxPython에 조금 의존할 수 있다.

---

## 대안 B. Queue + wx.Timer 사용

수신 스레드는 Queue에 메시지를 넣고, GUI는 일정 간격으로 Queue를 확인한다.

```text
수신 스레드
↓
Queue에 메시지 저장
↓
wx.Timer가 주기적으로 Queue 확인
↓
GUI 채팅창 갱신
```

장점:

- 네트워크 코드와 GUI 코드가 더 잘 분리된다.
- Tkinter, PyQt 등 다른 GUI로 바꾸기도 쉽다.

단점:

- 구조가 약간 더 길어진다.
- Timer 설정이 필요하다.

---

## 선택

본 프로젝트에서는 설계 분리를 위해 다음 방식을 사용한다.

```text
Queue + wx.Timer 방식
```

이 방식은 GUI와 네트워크의 책임을 분리하기 쉽고, 수신 스레드가 GUI 위젯을 직접 수정하지 않도록 만들 수 있다.

## 10. 메시지 프로토콜 설계

### 10.1 메시지 형식

메시지는 JSON 문자열로 구성한다.

일반 채팅 메시지:

```json
{
  "type": "chat",
  "sender": "hanjae",
  "message": "안녕하세요"
}
```

입장 메시지:

```json
{
  "type": "join",
  "sender": "hanjae",
  "message": "hanjae님이 입장했습니다."
}
```

퇴장 메시지:

```json
{
  "type": "leave",
  "sender": "hanjae",
  "message": "hanjae님이 퇴장했습니다."
}
```

### 10.2 메시지 구분 방식

TCP는 메시지 단위를 자동으로 구분하지 않는다.
따라서 본 프로젝트에서는 각 메시지 끝에 줄바꿈 문자 `\n`을 붙인다.

```text
{"type":"chat","sender":"hanjae","message":"안녕하세요"}\n
```

수신 측은 줄바꿈 문자를 기준으로 메시지를 분리한 뒤 JSON으로 변환한다.

## 13. 다른 컴퓨터에서 실행하기 위한 조건

서버는 외부 클라이언트 접속을 받을 수 있도록 다음 주소로 실행한다.

```python
HOST = "0.0.0.0"
PORT = 5000
```

클라이언트는 서버 컴퓨터의 실제 IPv4 주소로 접속한다.

예시:

```text
서버 PC IPv4 주소: 192.168.0.15
클라이언트 접속 주소: 192.168.0.15
포트: 5000
```

다른 컴퓨터에서 접속이 되지 않을 경우 다음을 확인한다.

```text
1. 서버와 클라이언트가 같은 와이파이에 있는지 확인
2. 서버 HOST가 127.0.0.1이 아닌 0.0.0.0인지 확인
3. Windows 방화벽에서 Python 실행을 허용했는지 확인
4. 포트 번호가 서버와 클라이언트에서 동일한지 확인
5. 학교 또는 공용 와이파이가 기기 간 통신을 차단하지 않는지 확인
```
